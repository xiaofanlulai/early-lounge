# :boom: ガイドライン
## 1. 命名規則
### クラス名
- BEM方式 ` block__element␣--modifier ` で統一。
- 複数語はキャメルケースで連結する。
- ~~TOPページはBlock名の頭に「top」を付け、blockに繋げる(キャメルケース記法)。~~
- modifierの後には要素を追加してはいけない。
- block名は同じものを使用しない。
```
block__element --modifier
例）company__image --active

company__imageWrapper --active

✕ content__inner --column--reverse
◯ content__inner --columnReverse
```
> [!CAUTION]
> elementとmodifierの間には半角スペースがあるので注意。
### id名
- 前提として、CSS適用の際のid指定は禁止。
- JavaScriptはidを定義して実装する。
- ` js-役割に基づく命名 ` で統一する。
```
js-toggleMenu
js-userProfile
```
### ファイル名
- 小文字とハイフンを使用。
- 意味のある名前を使用。
- 拡張子の一貫性 → .html css .scss .js
```
service.html
top.css
user-profile.js
```
### 変数名
- キャメルケースを使用。
```
userName
backGroundColor
```
### 画像ファイル
   - ` ページ名-セクション名-役割に基づいた名前.〇〇 ` で統一。
   - ~~ページごとにフォルダを分けない~~
   - 大文字は使用しない
   - 英単語は省略しない。
```
top-service-icon01.png
```

# :desktop_computer: GitHubガイドライン
## 1. ブランチ戦略
### ` main ` ブランチ
- 役割：デフォルトのメインブランチであり、常にデプロイ可能な最新コードを置く。
### ` develop ` ブランチ
- 役割：全ページの合流ブランチを統合するブランチ。エラーなどなく、` main ` ブランチと同じ最新なコードがあるのが理想。合流ブランチはここから派生する。
### ` ページ名-merge ` 合流ブランチ
- 役割：作業ブランチを統合するためのブランチ。` develop ` ブランチから派生し作成する。
### ` ページ名-markup-制作者名 ` 作業ブランチ
- 役割：各ページを実際に作業するブランチ。
## 2. コミットメッセージ
### サマリー
- ` カテゴリー：内容を一言 `の形に統一。
- カテゴリーは次の3項目から合うものを選択
```
feature：新しい機能や改善を追加した。
fix：バグ修正を行った。
docs：READMEやAPIドキュメント、コメントの追加・修正など、ドキュメント全般の変更を行った。
```
### 詳細説明
- 行ったことを箇条書きでわかりやすく簡潔に記載する。
### フッダー
- 基本的に記載なし
```
★ コミットメッセージの例 ★
【サマリー】
feature：headerのコーディング(レスポンシブ含め)
【説明文】
以下の変更を行いました。
・ナビゲーションメニューの背景色を変更し、アクセシビリティを向上
・メニュー項目のホバー時のアニメーションを追加
・スマホ表示時にメニューが適切に折りたたまれるようにCSSを修正
```
> [!IMPORTANT]
> 基本的に、コマンドプロンプトやターミナルでコミットする際は説明文とフッダーは不要。  
> 全く変更しなくてもコミットを行う必要がある場合は、` git commit -m "Update" ` でコミットを行う。
## 3. プルリクエストの定型文
### タイトル
- ` カテゴリー：内容を一言 ` の形に統一する。
- カテゴリーは次の3項目から合うものを選択
```
feature：新しい機能や改善を追加した。
fix：バグ修正を行った。
docs：READMEやAPIドキュメント、コメントの追加・修正など、ドキュメント全般の変更を行った。
```
### 説明文
- 「目的」「背景」「必要性」などを完結に書く。
```
★ プルリクメッセージの例 ★
【タイトル】
feature：headerの完成
【説明文】
以下の変更を行いました。
・ナビゲーションメニューの背景色を変更し、アクセシビリティを向上。
・メニュー項目のホバー時にアニメーションを追加して、視覚的なフィードバックを強化。
・スマートフォンやタブレットでの表示に対応するため、メニューの折りたたみ機能を追加。
・ヘッダー全体のレイアウトを調整し、コンテンツが各デバイスで適切に表示されるようにCSSを修正。
```
## 4. Gitフロー 各段階のタイミング
### ` 作業ブランチ ` のステージング＆コミット＆プッシュ
- タイミング：基本的に任意。作業が一区切りついた時点（例：新しい機能の完成が完了したときなど）で行う。
- ルール：
   - 1日の作業の終わりには必ずステージング＆プッシュ＆コミットを行う。
   - ステージング＆コミット＆プッシュは必ずセットで行う。
### ` 作業ブランチ ` → ` 合流ブランチ ` へのプルリクエスト
- タイミング：各セクション部分の実装や修正が完了(レスポンシブ含め)し、コードが安定していると判断された時点で行う。
- ルール：
   - プルリクエスト作成前にメンバー全体で動作確認を行い開発者同士でのレビューを行う。
### ` 作業ブランチ ` → ` 合流ブランチ ` のマージ
- タイミング：プルリク内容が承認されたら即座にマージする。
- ルール：
   - 全員のレビューが完了してからマージを行う（少人数のチームでは1人の承認でも可）。
### ` 合流ブランチ ` → ` developブランチ ` へのプルリクエスト
- タイミング：ページ単位でコーディングが完了した時点でプルリクエストを行う。
### ` 合流ブランチ ` → ` developブランチ ` のマージ
- タイミング：プルリクエストが承認された後、即座にマージする。
- ルール：
   - マージ後、動作検証を実施し問題がないか確認する。
### ` developブランチ ` → ` mainブランチ ` へのプルリクエスト
- タイミング：` developブランチ ` に全ページが統合され挙動確認が通ればプルリクエストを作成する。
### ` developブランチ ` → ` mainブランチ ` のマージ
> [!IMPORTANT]
> ` main ` ブランチには直接プッシュを行わず、必ず` develop ` ブランチからプルリクエストを通じてマージするように！
